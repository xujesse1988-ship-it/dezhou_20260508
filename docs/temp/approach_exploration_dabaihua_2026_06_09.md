# 6-max 方案重审（大白话版，2026-06-09）

## 这份文档是什么

不被现有执行文档「实时搜索是主干」框死，回到目标本身，重新问一句：**要在真实 6-max 牌局里尽量打近最优，到底有哪几条可行的路？** 网上调研了一圈别人怎么做（Pluribus / ReBeL / 商业 solver / 各种 CFR），又把咱们自己代码的现状核查了一遍，最后列了 7 个候选、排了名、给了推荐。

这是一份探索记录，不是拍板的执行计划。结论可能被后面的实测推翻——尤其是「实时搜索在深码多人树上到底解不解得动」这条命门，现在还没数。

## 目标先说清楚

真实牌局有三个绕不开的维度：

- **人数**：见 flop 可能 2 人，也可能 4-5 人。
- **码深**：各家起始码深不相等、连续变化，14BB 短码到 800BB 深码都有。
- **限时**：每步决策 5/10/20 秒。

计分是 chip-EV、无 rake（对齐 OpenPoker 那个场子）。注意「最优」在多人非零和博弈里**没有 Nash 均衡保证**，所以这里说的「最优」实际意思就是：**解出一个稳健、自己跟自己不打架（接近不动点）的策略**，而不是一个有数学保证的东西。

## 一个重新框定：便宜的活和难的活不在一处

把三个维度拆开看，会发现**好下手的地方和真正难啃的骨头根本不在一处**：

### 码深维度其实有现成抓手

规则引擎**本来就支持任意不对称码深**——`starting_stacks` 是 per-seat 的 `Vec`，side-pot、payout、重发隐藏牌对任意人数任意栈都算得对（`state.rs`、`config.rs`）。真正把策略锁死在 100BB 的，是**抽象表那一层**：`pack_info_set_v2` 把 `stack_bucket` 这个字段硬编码成 0（`nlhe.rs:123`，注释写着「起手筹码固定」），所以策略对码深完全瞎。

换句话说，「现场按真实码深建一棵真树、现场解」这条路对码深维度**机制上是现成可用的**，引擎不用动，瓶颈只在两处上游：① 真实码深从入口就没喂进来；② blueprint 的对手 range 先验还是按 100BB 算的。

### 人数 4+way 才是真正的硬骨头

这个维度**没有便宜解**：

- 重训一个 4-way blueprint，几乎确定会因为「同样的训练预算摊到更大的表上 → 覆盖率太低 → 欠训练」而变得更差（详见后面专门一节）。
- 实时解一棵真的 4+way 子树要多久，**从来没在目标树上测过**。
- 连 Pluribus、GTO Wizard AI 这种公开/商用最强的，也只让 3 人见 flop，靠 preflop 折叠把 5-6 人压掉。4+way 的 postflop 全行业都还没真解。

### 限时这块连地基都没打

求解器**现在根本停不下来**：`subgame.rs` 的求解循环是 `for _ in 0..iterations` 固定迭代数，全文件没有任何 deadline。它还不是「解到时间用完就返回」的那种随时可停的求解器，没法直接塞进 5/10/20 秒的时限里用。

## 7 个候选方案 + 排名

| 排名 | 方案 | 分（满分10） | 一句话 |
|---|---|---|---|
| 1 | **F 务实接线**（anytime + 喂真栈 + 正确性硬验收） | 9 | 风险最低、几乎纯接线，三个缺口都核实属实且没被任何实验证伪。是一切攻坚前该先有的地基 |
| 2 | **A 实时子博弈搜索为主干** | 7 | 唯一原理上能覆盖「连续码深 × 人数」、又复用了约 85% 现有 Rust 代码的在线路子；但「主干」言过其实——目标树上的耗时全没测、已测的格子里点估全是负的 |
| 3 | **G 离线自洽性判据** | 6 | 本身不提升强度，但「正确性大于一切」要求它。瘦身后只留两件几乎免费的 |
| 4 | **C 按码深分桶的多 blueprint 族** | 5 | 对码深维度能硬证改进、跟行业（PioSolver 每个 SPR 重解一遍）对齐，但只解一维、不解多人，而且基本被 A 覆盖 |
| 5 | **E 商业 solver 范式**（preflop 大表 + postflop 现场解） | 4 | postflop 那一半逐字等于 A，独立价值最小；它引以为据的「行业范式」其实靠的是神经网络，咱们没有 |
| 6 | **B 重训真正的 4-way blueprint** | 3 | 点名候选，但证据强烈不支持作主干（专门一节讲） |
| 7 | **D 神经网络**（ReBeL/SoG 值网 或 Deep CFR） | 2 | 多人是它的死穴不是优势，成本高 2-3 个数量级，还得纯 Rust 从零搭 |

### 每个候选简单说

- **A 实时搜索**：每次决策按真码深、真人数、真 board 现场建一棵子树，用 CFR 现场解，blueprint 退成「先验 + 叶子续局值 + 兜底」。好处是这是**唯一不组合爆炸、能覆盖连续码深×人数空间**的办法，而且复用了约 85% 现有代码。坏处是它的关键数据（深码多人树上单决策要多久、能解几迭代）全没测，而且已经测过的小格子里点估都是负的。

- **B 重训 4-way blueprint**：把 postflop 收口从 ≤3-way 放宽到 ≤4-way 重训一张表。见下面专门一节。

- **C 码深分桶 blueprint 族**：preflop 走连续翻译，postflop 把进 flop 时的 SPR 分成几档（比如 4-6 档）各训一张表，覆盖码深维度。跟行业「每个 SPR 重解」的主路对齐，但只解码深不解多人，而且 A 的「现场建真树」本来就天然覆盖码深，所以 C 基本被 A 支配，顶多当 A 的离线加速兜底。其中「在一张表里加码深维度」（C2）要动核心索引、破掉所有 byte-equal 校验锚，风险最高，建议砍掉。

- **D 神经网络**：用一张神经网络把「栈向量 + 人数 + board + range」当输入，泛化整个连续空间。问题是：① 多人是死穴——ReBeL / Player of Games / Student of Games 的作者都亲口把理论保证限定在两人零和，N>2「意义不大、效果不明」；② 成本高得离谱（ReBeL 单挑就用了 720 块 V100、约 45 亿样本，咱们零 GPU）；③ 它相对查表的唯一实打实好处是码深泛化，但单挑实测也只能打平好的抽象、并不更强。不该当主干，最多以后单独拿来做「码深条件化的叶子值估计」。

- **E 商业 solver 范式**：preflop 按码深预解一大堆 chart，postflop 现场 endgame solve。但 postflop 那一半就是 A，没有独立价值；而且它「行业验证」的强度其实来自神经网络叶子估值，咱们没有，这个背书不成立。

- **F 务实接线**：见第 1 名，下面推荐路径就是它打头。

- **G 离线自洽性判据**：核心区没有干净的离线 EV 标尺（多人非零和 exploitability 失效），G 想补一套能算的弱判据。瘦身后值得做的只有两件、且几乎免费：① **被支配动作扫描**（线性扫一遍表，专抓实现 bug）；② **单边 best-response gap**（固定其余人、只让一人 best-respond，量自对弈稳不稳）——而且这个其实 `lbr.rs` 已经是它的采样实现了，并进「LBR 推广到 N 座」就行，不用新写。关键是文档里必须钉死：**这个量度的是「自己跟自己稳不稳」，不是「绝对有多强」**，别被误读成「这个数低 = 强」。

## 推荐路径（分两阶段）

### 第一阶段：1-2 周，全在 vultr 上跑，零 AWS 预算，无悔投资

这一阶段**不承诺「搜索就是主干」**，只干两件事：让 bot 第一次能端到端跑真实分布 + 把那条命门数据测出来。本质就是 F + 一次命门耗时实测。

1. **接 anytime + 接 LCFR**（半天）：给求解循环加个 `time_budget`、改成解到 deadline 就停；顺手把机器里已经写好的 `with_lcfr_period` 接进子博弈求解循环（现在白扔着没用）。
2. **喂真栈 + 接进生产**（半天）：`Request` 结构加 per-seat 栈字段，`decide()` 用真栈构 `TableConfig`，让搜索从带真栈的 template 建真 SPR 树；并且把搜索**真正接进生产 `decide()`**（现在线上是纯 blueprint 查表，搜索一次都没被调用过）。
3. **命门实测**（1-2 天）：在**真正的目标树**上——4+way 不收口、加上不对称深栈（比如 hero 200BB 对手 60BB）、加上多人 side-pot 的中途局面——实测「节点数、迭代数 → 单决策耗时」+ 桶 fallback 率 + 收敛曲线。**这一步直接回答「5/10/20 秒内深码 4+way 解不解得动」这个全方案的生死命门。**
4. **顺手把 4-way 事实化**（几小时）：跑一次 `nlhe_betting_tree_sizing --postflop-cap 4`，把 4-way 的「48GiB / 14.45 亿 infoset」从文档估算变成真实测量，再在那棵真 4-way flop 子树上测一下单决策耗时。这正是「4-way 值不值得」的对照数据。
5. **复核旧负判决**（1 天）：之前那些「搜索不帮忙」的负结论（range-on −71.6、all-postflop −192 mbb/g）全是在 **1B / 56% 覆盖的弱 blueprint** 上测的；现在有了 **10B / 82.55% 覆盖**的强 blueprint，用它把那个 A/B 复跑一遍，看符号变不变。

第 1-2 步让 bot 能跑真实分布，第 3-4 步给出「接着该投多少」的决策数据，第 5 步复核旧结论该不该翻案。全部 vultr 能做、不用训练、不用申请大机。

### 第二阶段：按第一阶段的耗时结果分叉

- **如果深码 4+way 在时限内解得动、fallback 率也可控** → 把叶子续局值和对手 range 先验**按真码深重标定**（这是硬前置，不是「以后优化」），然后接 OpenPoker 当弱护栏上线。
- **如果解不动** → 用 C1 预存几个码深档的 blueprint 当兜底加速层；或者把时限收窄到 10-20 秒 + 申请 ≥8 核确认。

### 明确不投

- 神经网络转向（D）——多人没优势、成本高得离谱。
- 在单张表里加码深维度（C2）——破掉所有 byte-equal 校验锚、动核心索引、风险最高。
- E 当独立主干——postflop 那半就是 A，没独立价值。

## 专门说说「重训 4-way blueprint」

这是点名要评估的候选。直接说结论：**不值得作为主干去真训，但值得花几小时把它从「估算辩论」变成「实测对照」再下最终结论。** 三个硬理由：

1. **只碰一个维度，码深完全没碰**：它把人数从 ≤3-way 推到 ≤4-way，5-6 人见 flop 照样收口。码深维度它完全无能为力（`stack_bucket` 恒为 0）。
2. **覆盖率几乎确定灾难**：N=3 那张表跑到 10B 次更新才到 82.55% 覆盖；N=4 是 6.27 倍节点，**同样 10B 预算只能买到大约 11-13% 覆盖**。要追平 N=3 的训练密度得约 63B 次更新 ≈ 8 天 ≈ 250 美元 AWS。更糟的是 4+way 这条线本来就最少见、采样最稀，**你想加价值的那部分恰恰训得最差**。
3. **就算训出来也没干净尺子验它强不强**：多人非零和没有 exploitability，OpenPoker 又不能配对、功效低，欠训练的 4-way 在 4+way 局面还更难评。

**内存根本不是瓶颈**（48GiB 两张表，c6a.8xlarge 的 64GiB 勉强放得下）——覆盖率 + 码深零覆盖 + 没标尺才是。

正确的顺序是：真要推多人维度，得**先**用 raise cap + imperfect recall 把每一路的 infoset 砍掉一个数量级，**再**考虑扩到 4-way，否则覆盖率天花板直接锁死。同样的算力花在「实时搜索喂真栈 + 真人数」上对目标回报高得多——因为引擎本来就支持任意不对称码深，而 4-way blueprint 对码深维度一点忙都帮不上。

## 调研里几条值得知道的新发现

1. **LCFR 被白扔了**：机器里 `with_lcfr_period` 已经实现了（`trainer.rs:332`），但子博弈求解循环用的是裸的 `EsMccfrTrainer::new`，**没接 LCFR**。接上去就几行改动，可能直接改善「5 秒内能解几迭代」这个命门。（提醒：综合阶段曾说「DCFR/CFR+ warm-start」，那是凭空想的——真求解器是 ES-MCCFR、没有 warm-start；但 LCFR 确实存在、只是没接，这是真的。）
2. **旧的负判决全是在弱 blueprint 上测的**：所有「搜索不帮忙」的数字都跑在 1B / 56% 覆盖的弱表上。现在有 10B / 82.55% 的强表，文档自己都写它「远强于」当时——旧结论该当「过时读数」复核，不该当定论给搜索判死。
3. **核心区强弱这套基建自己验收不了**：「search-on 一边 vs 同一个 blueprint 当对手」这种自对弈探针，**结构上就测不出绝对强弱**（blueprint 越强、当对手越硬，自对弈偏移亏得越多）。这不是能绕过去的 confound，是探针的本性。意思是：绝对强弱只能靠**外部对手**（OpenPoker 弱护栏），离线只能硬证「解的是真游戏」。这点要正面接受，不是退而求其次。
4. **兜底不能回落 100BB blueprint**：解不动就回落 blueprint，等于把「解真游戏」打回「解错游戏」，正是想修的毛病。兜底动作得来自**真实状态的启发式**，而这层 `src/` 里现在不存在，要一并建。

## 该问用户的几个问题

1. **OpenPoker 的 api_key 拿到了吗？那个场子里对手池真实的码深 / 人数分布大概什么样？** 这决定先做哪个码深档、4+way 到底多常见、弱护栏功效有多少。
2. **愿不愿意先做第一阶段第 1-4 步**（全 vultr、零训练预算、约一周）——它能把「4-way 值不值得」和「深码 4+way 解不解得动」从嘴上辩论变成有数可比。
3. 第 3 步命门实测如果发现深码 4+way 严重超时，**愿不愿意申请 ≥8 核**按预算确认？

## 老实交代几处没验 / 不确定的

- **4-way 的「48GiB / 14.45 亿 infoset」严格说还是文档估算**。调研里有个 agent 声称它「实跑」了 sizing 并得到精确吻合的数字，但正因为太吻合、我没把它当成已验证的新测量（很可能是把文档估算包装成了实跑）。所以第一阶段第 4 步专门要真跑一次把它钉死。
- **单决策耗时在目标树上是真空白**。已有的「24k 手 / 95 秒 / 4 核」是在小的 100BB ≤3-way flop 子树上测的，外推到深码 4+way 站不住——深码 SPR 更高、树更深，节点数会涨；4 核够不够全看实测。
- **F 不等于「码深已覆盖」**。喂真栈后「现场建真树」只在短码小树（能解到终局）那档干净成立；真正的深码 150-800BB 树太大解不到终局，必须靠 depth-limit + 叶子续局值，而那个叶子值表现在还是 100BB 的——深码叶子值重建是 v1 的核心硬活，不是「以后重标定」能糊过去的。
