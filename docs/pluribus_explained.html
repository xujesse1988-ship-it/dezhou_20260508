Pluribus 算法深度解读

从论文到 Rust 实现 · 6-max 无限注德州扑克 AI 的工程化全景

Pluribus（CMU + Facebook AI，2019 Science 论文）是第一个在 6 人桌无限注德州扑克（6-max NLHE）中战胜顶级人类职业玩家的 AI。在与 12 名世界级职业选手的 10,000 手对局中，Pluribus 以统计显著的优势赢钱——而它训练所用的"自博弈"成本只有大约 150 美元的云算力。

这份文档不是论文复述。它把 Pluribus 的核心思想拆成可工程落地、可量化验收的模块，并对应到本仓库 dezhou_20260508 的实际 Rust 代码与 8-stage 路线。读完之后，你应该能回答两个问题：它在算法上做对了什么？它在工程上又如何把这些算法塞进现实的算力预算？

1. 难点：6-max NLHE 为何如此难

德州扑克之所以让 AI 头疼，不是因为某一项困难，而是它把博弈论里几乎所有"硬"问题同时叠在了一起：

① 不完美信息

对手的底牌不可见。AI 必须在信息集（infoset）上推理：所有"我看到的相同公共历史 + 我自己的底牌"的世界状态被合并成一个决策点。

② 巨大的状态空间

6 玩家、4 条街、52 张牌、连续筹码量。原始游戏树节点数估计 ~10165，远超国际象棋 ~1047、围棋 ~10170，且不完美。

③ 多玩家

2 人零和有理论上的 Nash 均衡保证；≥3 人的零和（或一般和）博弈不再有这种保证。即便算出某种"均衡"，对手仍可能合谋偏离把你吃掉。

④ 连续动作空间

"加注多少"是一个连续量：理论上可加 1 chip、2 chip、… 整个动作集合是 [2BB, stack] 的实数区间。CFR 这类算法只能在有限动作集合上运行。

关键数字对比

国际象棋（DeepBlue 1997 / AlphaZero 2017）和围棋（AlphaGo 2016）都是 2 玩家完美信息。HU NLHE（2 玩家无限注）于 2017 年被 Libratus 攻克。6-max NLHE 直到 Pluribus（2019）才被攻克——这正是博弈论 AI 的"最后一公里"。

3. CFR 与 Regret Matching：自博弈的引擎

3.1 直觉：什么是"后悔"？

想象你打完了一手牌：在某个决策点你 check，最终输了 50 chip。事后看，如果你当时改 raise，可能赢 80 chip。那么这个决策点上 raise 相对 check 的"后悔值"就是 +130。

Counterfactual Regret Minimization（CFR）的核心想法非常朴素：

CFR 一句话总结

对每一个信息集（infoset）、每一个动作，累积这种"如果当初改选这个动作能多赚多少"的反事实差值。下一轮自博弈时，按"正后悔"成比例挑动作。重复几十亿次，平均策略收敛到 ε-Nash 均衡。

3.2 公式

设信息集为 I，动作集合 A(I)，迭代次数 t。每个 (I, a) 对累积两个数：

$$ R_t(I, a) = \sum_{\tau=1}^{t} \pi^{-i}_\tau(I) \cdot \left[v^\tau(I, a) - v^\tau(I)\right] $$

其中 $\pi^{-i}(I)$ 是对手与机会节点到达 I 的概率（counterfactual reach probability，CFR 名字的来源），$v^\tau(I,a)$ 是第 τ 轮 traverser 视角下 I 上选 a 的期望价值，$v^\tau(I)$ 是当前策略 σ 的期望价值。

下一轮 σ 由 Regret Matching（Hart & Mas-Colell, 2000）给出：

$$ \sigma_{t+1}(I, a) = \frac{\max(R_t(I,a), 0)}{\sum_{b \in A(I)} \max(R_t(I, b), 0)} $$

当分母为 0（所有 regret ≤ 0），退化为均匀分布。

最终用 average strategy（不是 current strategy！）作为输出——这是 CFR 收敛性的关键：

$$ \bar\sigma_T(I, a) = \frac{\sum_{t=1}^T \pi^i_t(I) \cdot \sigma_t(I, a)}{\sum_{t=1}^T \pi^i_t(I)} $$

3.3 在代码里长什么样

本仓库 src/training/regret.rs 里的实现非常直接：

/// Regret Matching：σ(I, a) = max(R(I,a), 0) / Σ max(R(I,b), 0)
/// 全部 regret ≤ 0 时退化为均匀分布（D-303 / D-331）
pub fn current_strategy(&self, info_set: &I) -> SigmaVec {
    let regrets = match self.inner.get(info_set) {
        Some(r) => r,
        None => return SigmaVec::new(),
    };
    let pos_sum: f64 = regrets.iter().map(|&r| r.max(0.0)).sum();
    if pos_sum > 0.0 {
        regrets.iter().map(|&r| r.max(0.0) / pos_sum).collect()
    } else {
        // 退化均匀分布
        let n = regrets.len() as f64;
        regrets.iter().map(|_| 1.0 / n).collect()
    }
}

整个 CFR 的"灵魂"就这 5 行。所有复杂性都在如何在合理时间里把这 5 行调用足够多次。

4. MCCFR：采样让 CFR 跑得起来

Vanilla CFR 每轮需要遍历整棵博弈树。德州游戏树太大，整棵跑一遍要几年。MCCFR（Lanctot 2009）的关键观察：

采样替代遍历

每轮不必遍历整棵树。只采样一条 trajectory，按重要性重新加权 regret 更新。期望收敛速度从 O(全树/iter) 变成 O(单条 trajectory/iter)，每秒 update 数提升几个数量级。

4.1 External-Sampling MCCFR（ES-MCCFR）

Pluribus 用的是 External Sampling 变种（也是本仓库 Stage 3/4 选型，详 D-301）：

  • 选定本轮的 traverser（视角玩家）。
  • traverser 自己的决策点：遍历所有 action，分别递归计算每个分支的 cfv（counterfactual value）。这是"全展开"的方向。
  • 对手 + 机会节点（发牌）：采样一个 action（按当前策略概率），只递归这条分支。这是"采样"的方向。

这样 每条 trajectory 走完 ≈ 一次 update，traverser 视角下的 regret 表得到完整更新，而对手视角的 regret 留到下一轮当他做 traverser 时再更新。6 玩家就轮转 6 个 traverser（D-307 alternating，本仓库 NlheGame6 实现）。

图 2：External-Sampling MCCFR — 一次"展开 + 采样"组合，比 Vanilla CFR 快几个数量级。

本仓库 EsMccfrTrainer::step 在 src/training/trainer.rs 里实现这套逻辑。每次 step() 调用 = 1 个 traverser 的 1 次 sampled update。10¹¹ update 就是这种 step 跑这么多次。

5. Linear MCCFR + RM+：两个让自博弈"早收敛"的小改动

朴素 MCCFR 收敛率是 O(1/√T)。Pluribus 用了两个工程上看似很小、却让收敛速度显著加速的 trick：

5.1 Linear Weighting（线性加权）

把新累积的 regret 权重 ↑，旧累积的权重 ↓。具体做法：每轮 t 起始时给已有 regret 乘上一个衰减因子：

$$ R_t(I, a) \leftarrow R_{t-1}(I, a) \cdot \frac{t}{t+1} + \text{(本轮新增)} $$

直观：自博弈早期策略很烂、产生的 regret 噪声很大，应该被后期更高质量的 regret 覆盖。Linear 是 Pluribus 实测最优的"平滑老 regret"形式（其它如 CFR+ 用阈值截断、Discounted CFR 用指数衰减，效果各有千秋）。

5.2 Regret Matching+（RM+）

每次更新 regret 后立即把负值截断到 0：

$$ R_t(I, a) \leftarrow \max\left(R_t(I, a), 0\right) $$

直观：负 regret 长期积累后，要让一个"曾经很烂"的动作被重新选中，得花很多次 update 才能把累积值拉回 0。RM+ 强制每次重置，等价于给坏动作一个二次机会。论文与本仓库实测在多人 NLHE 上 RM+ 收敛比纯 RM 快约 2-4×。

5.3 在代码里

本仓库 src/training/trainer.rs 中 EsMccfrTrainer::apply_decay + clamp_nonneg 实现这两步（D-401 + D-402）：

// 每 iter 起始：扫全表应用 t/(t+1) decay（D-401 EagerDecay 主路径）
fn apply_decay(&mut self, t: u64) {
    let factor = (t as f64) / ((t + 1) as f64);
    for regrets in self.regret_table.iter_mut() {
        for r in regrets.iter_mut() { *r *= factor; }
    }
}

// 每次 regret update 后：RM+ floor at 0 （D-402）
fn clamp_nonneg(regrets: &mut [f64]) {
    for r in regrets.iter_mut() { *r = r.max(0.0); }
}
Warm-up 期

Linear MCCFR + RM+ 在 t=0 附近会过度抑制初始 regret，导致策略卡在均匀分布。本仓库的解决方案（D-409）：前 1M update 走标准 CFR + RM，之后再切换到 Linear + RM+。等价于"先让 regret 长出来，再开启加速器"。

6.1 量化公式

raise 金额公式（compute_raise_to）：

$$ \text{raise\_to} = \text{current\_bet} + \lfloor \text{multiplier} \times \text{pot\_size} \rfloor $$

floor 舍入是字面选择（B2 测试容差 ±1 chip 允许 floor/round/ceil 任一）。

训练过程中实时输出三组指标（D-470）：

  • Average regret 增长曲线：理论应 sublinear（O(√T)），线性增长意味着算法在发散。
  • 策略 entropy：早期高（接近均匀），后期下降但不应坍缩到 one-hot（坍缩说明 RM+ 卡在某分支）。
  • 动作概率震荡幅度：训练后期振幅应衰减，持续大幅震荡 → 报警。

9. 实时 Depth-Limited Search：blueprint 的"现场校正"

Blueprint 是抽象博弈上的近似 Nash。直接拿它打实战会撞上两个问题：

  1. 抽象动作集合（14 个）和真实下注金额不匹配 → off-tree 问题。
  2. 对手可能在抽象边界附近"卡位"剥削。

解决方案：每个决策点从当前 public state 出发，做一次有限深度子博弈重解，叶子节点用 blueprint 估值，输出更优的本地策略。这就是 Pluribus 的 "continual re-solving"（stage 6a）。

图 4：Depth-Limited Search — 不再用整棵树，只在 P95 < 30 秒预算内求解当前子博弈。

关键技术点（path.md 阶段 6a）：

  • 当前下注轮用更细的 action abstraction（甚至接近 lossless），未来下注轮用受控 bucket。
  • 叶子节点用 blueprint 估值。这避免重新搜索整个未来。
  • subgame 总 action sequence 控制在 100-2000，否则求解爆炸。

10. Biased Leaf Strategies：Pluribus 真正能赢人的关键

这是 Pluribus 论文最难理解、也最有创意的部分

它解释了为什么 Pluribus 不需要建模对手，也能在多人桌上稳赢——靠的是在自己的子博弈解里假设对手有 4 种 meta-strategy 可选，给"被剥削"留出明确的应对。

10.1 问题：朴素 search 会被反剥削

朴素方案：每个决策点 solve 一次子博弈，叶子直接用 blueprint 估值。问题是—— blueprint 是对全局近似 Nash，到了某个具体 subgame 它可能过于一致。聪明对手知道你后面会按 blueprint 出牌，就可以选择对你最不利的偏离。

10.2 Pluribus 的解法：4 个 Biased Leaf 策略

在子博弈叶子节点，不止给一个 blueprint 估值。而是给对手 4 个可选的 meta-strategy：

① Unbiased

对手按 blueprint 原样出牌。

② Fold-Biased

对手系统性地更倾向 fold（叶子估值用"对手多 fold 时"的 EV 计算）。

③ Call-Biased

对手更倾向 call（被动跟注，骗信息）。

④ Raise-Biased

对手更倾向 raise（激进试探）。

子博弈求解时假设对手会选择 4 个策略中对 ta 最有利的那个。这相当于在 subgame 解里把"对手会动手脚"显式建模成 max-min。Pluribus 的策略就是在这个更悲观的假设下做出反应。

为什么这破解了"无法建模对手"的难题

传统 opponent modeling 要观察对手很多手才能判断 ta 是哪种类型；Pluribus 不建模具体对手，而是预设对手"理论上"可以选这 4 种偏离。如果它的策略对这 4 种都不亏，那对任何介于其间的真实人类对手也不会亏太多。这是纳什的稳健化（robustification）。

每条 biased strategy 都能单独消融开关。论文与本仓库 stage 6b 都要求：去掉任一 biased leaf 后可剥削性显著上升，证明这 4 条不是装饰。

11. Off-tree Action Handling：边界剥削防御

抽象动作集合是 14 个。如果对手 raise 真实金额刚好夹在两个抽象动作之间（比如他 raise 0.625 pot，介于 0.5 pot 和 0.75 pot 之间），Pluribus 必须把这个动作映射到某个抽象动作上，否则没法查 blueprint。

11.1 Pseudo-Harmonic Mapping

Pluribus 用 pseudo-harmonic mapping（Ganzfried & Sandholm, 2013）。直觉：

  • 真实下注 x 落在抽象动作 A（金额 a）和 B（金额 b，a < x < b）之间。
  • 用调和平均而不是算术平均，因为下注金额的对数空间更接近策略意义上的"远近"。
$$ P(A \mid x) = \frac{(b - x)(1 + a)}{(b - a)(1 + x)} $$

剩余概率给 B。然后随机化选 A 或 B，避免被对手通过"边界偏移"剥削。本仓库占位实现在 src/abstraction/action_pluribus.rs::map_off_tree（stage 6c 完整实现）。

最大的实战 bug 来源

Pluribus 论文与本仓库 path.md 都强调：off-tree handling 是实战阶段 bug 与可剥削性的最大来源。stage 6c 必须 fuzz 1M 个 off-tree 下注金额、对故意卡边界的对手做 1M 手对战，统计是否能通过卡边界获得显著正收益（视为可剥削性证据）。

12. 为什么 Pluribus 能击败职业玩家

所有零件都摆完了。回到最开头的问题：这一套东西凑在一起，为什么真的能赢顶级人类？

12.1 三个互相强化的论点

论点 ①：纳什均衡近似在不完美信息下是稳健的

2 人零和有理论保证：打 Nash 策略不会输（期望意义下）。6 人非零和虽然没有这种保证，但 Pluribus 用 6-traverser self-play + 4-biased leaf 把"被剥削的最大可能损失"压到了人类难以察觉的小量级。LBR exploitability < 100 mbb/g 意味着即便神级对手知道 Pluribus 的全部策略，每 100 手最多只能赢 1 个大盲。

论点 ②：不依赖对手建模 → 不可被"反扮演"

很多扑克 AI 试图建模对手风格（tight / loose / aggressive）。问题是对手知道你在建模 ta，可以反向欺骗。Pluribus 完全无视对手是谁，每个决策点只看公共历史 + 自己底牌，按 blueprint + search 出牌。结果是对手没法通过演戏诱导 Pluribus 走入陷阱。

论点 ③：实时 search 弥补了抽象的粗糙

Blueprint 是 ~10⁹ bucket 上的近似策略，单看挺粗糙。但每个决策点重新 solve 一次 subgame，把当前真实金额、真实公牌带入精细求解，相当于"用 blueprint 当估值函数，做一个浅层的 AlphaZero-style 搜索"。Pluribus 论文报告：去掉 search 后胜率从 +5 bb/100 降到接近 0。search 不是锦上添花，是必需的。

12.2 一个常见误解

"Pluribus 能赢，是因为算力大"？

不完全对。Pluribus blueprint 训练只用了 ~150 美元云算力，real-time search 用一台 28 核机器。同一时期的"算力堆栈"做法（暴力扩大抽象 + 长时间 CFR）都没能达到 Pluribus 的水平。Pluribus 的优势是算法选型（Linear MCCFR + biased leaf + depth-limited search），算力只是把这些算法跑完所必需的工程预算，不是优势来源本身。

13.2 关键不可妥协的工程约束

  • 无浮点进入规则/评估/历史/抽象路径。Chips 是 u64，P&L 是 i64。浮点只允许出现在 training::regret 内部（CFR 必须）。
  • 无全局 RNG。所有随机性走 RngSource 显式参数。便于复现 + 跨架构对齐。
  • 无 unsafe 代码。Cargo.toml 编译期 unsafe_code = "forbid"。
  • 决定性 baseline：同架构 + 同工具链 + 同 seed → 同 hand history BLAKE3 hash。跨 commit byte-equal 维持。
  • 测试与实现严格分离：[测试] agent 不动产品代码，[实现] agent 不动测试。所有越界都有 D-NNN-revM carve-out 显式追认。

结语

Pluribus 的胜利不是"AI 突然变聪明了"。它是 30 年博弈论积累 + 10 年 CFR 工程经验 + 几个关键工程小 trick 的合流：

  • CFR（Zinkevich 2007）是基础引擎。
  • MCCFR（Lanctot 2009）让它跑得起。
  • Linear MCCFR + RM+ 让它收敛得快。
  • Action / Information abstraction 让它装得下。
  • Depth-Limited Search + Biased Leaf Strategies 让它能赢人。
  • Off-tree mapping 让它不被边界剥削。

本仓库的工作就是把这条路线一砖一瓦用 Rust 重写，每一步都有量化验收门槛、test-first 流程、byte-equal baseline。目前刚刚走完 Stage 4 主算法落地，下一步是 F1 [测试]：钉死 Slumbot 评测协议 + 三类 baseline + cross-host 锚点，为 production blueprint 训练做准备。

真正的难关——stage 6 的 continual re-solving + biased leaf——还在前方。但因为前面 5 个 stage 把抽象、blueprint、checkpoint、监控都做扎实了，stage 6 的工程难度会显著降低。这就是 8-stage 串行的意义：不抄近路、每一步都可量化验收，最终在合理算力下复现"AI 击败人类职业玩家"这一里程碑。
